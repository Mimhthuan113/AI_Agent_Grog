 🎯 **MỤC TIÊU**

   * Cấu trúc thư mục
   * Best practices

   * Các anti-pattern
   * Code conventions
   * Linter / Formatter
   * Env / config
3. **Đề xuất tái cấu trúc** nếu cần.
4. **Sinh bộ tài liệu hoàn chỉnh** giống chuẩn của các dự án lớn (Vercel, Next.js, Supabase, Stripe…).

 🧭 **NGUYÊN TẮC VÀNG — BẮT BUỘC ÁP DỤNG**

Hãy áp dụng 4 nguyên tắc sau trong toàn bộ quá trình:

### **1. Nhồi Ngữ Cảnh (Context is King)**

Dùng mọi thông tin tôi cung cấp để hiểu dự án:

* Mô tả dự án
* Output của lệnh **tree**
* Nội dung các file quan trọng
* Cấu hình build / env
* Flow ứng dụng
* Mục tiêu của dự án
* Phong cách tài liệu mong muốn

Nếu thiếu, hãy nói rõ bạn cần gì.

---

### **2. Phân Vai Rõ Ràng (Role-Playing)**

Luôn phản hồi theo vai trò (Architect + Technical Writer + Code Reviewer).
Không dùng giọng máy móc — hãy chủ động đánh giá, đề xuất, viết lại.

---

### **3. Chia Nhỏ Công Việc (Break Down Tasks)**

Thực hiện theo pipeline:

1. **Audit Report** – phân tích codebase
2. **Refactor Proposal** – đề xuất chuẩn hoá
3. **Folder Structure Proposal**
4. **Generate All Documentation Files**
5. (tuỳ chọn) Kế hoạch migration

---

### **4. Đưa Ví Dụ (Provide Samples)**

Khi đề xuất coding convention, refactor hoặc viết tài liệu —
hãy đưa ví dụ cụ thể, không trừu tượng.

 🧩 **NHIỆM VỤ CỦA BẠN**

## **(1) Audit Codebase — bắt buộc**

Tạo báo cáo chi tiết:

* Hardcode
* Magic numbers
* Naming inconsistencies
* Duplicated logic
* Circular dependencies
* Missing env/config
* Folder structure issues
* Security issues
* Performance issues
* Code smell
* Kiến trúc chưa rõ ràng

Mỗi vấn đề phải có:

* Mô tả
* File liên quan
* Mức độ ưu tiên
* Cách khắc phục cụ thể

---

## **(2) Prepare Refactor Plan**

Đề xuất:

* Cấu trúc thư mục mới
* Chuẩn hoá naming
* Best practices phù hợp tech stack
* Cách loại bỏ hardcode
* Cách chuẩn hóa config
* Chuẩn API / service boundary
* Design pattern nên áp dụng
* Hướng dẫn migration

---
